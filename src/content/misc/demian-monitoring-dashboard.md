Demian의 모니터링과 대시보드에 포함될 수 있는 영역을 세분화해야한다.
현재 시스템은 개별 기능 구현 관점의 설계 논의는 많이 진행되었으나, 전체 시나리오와 관리 측면에 대한 논의가 부족하다고 느낀다. 이에 기본적인 시나리오들을 나열하고, 이에 따른 잠재적 니즈를 파악한 뒤에 각 니즈를 사용자 페르소나 별로 구분한다. 사용자 페르소나는 어드민(보안팀; 또는 데미안 개발팀)과 외부 개발팀 또는 PM이라고 제한한다.

## 시나리오

#### **1. 주요 시나리오 및 흐름**

1. **패키지 빌드에 의한 GitLab Webhook**
    - GitLab CI가 패키지 빌드를 완료하면, Demian 파이프라인 시작을 위한 Webhook 호출
    - Admin 관점: 파이프라인 모니터링·제어 뷰에서 신규 Job 생성 알림 필요

2. **신규 취약점 발견에 의한 Trigger**
    - SCA/SCE 스캔 결과 Critical·High 취약점 감지 시점에 알림 및 후속 워크플로우 시작
    - Admin 관점: 즉시 Jira 티켓 자동 생성·할당, Slack 알림 필요

3. **작업 시도 실패에 따른 재시도**
    - 네트워크 오류 등으로 개별 Job 실패 시, 수동/자동 재시도 트리거
    - Admin 관점: 실패 작업 카드에서 “재시도” 버튼 제공, 재시도 상태 추적 필요

4. **GitLab Pipeline 최종 Stage의 Report 링크**
    - 최종 Stage에 생성된 Report 링크를 통해 Demian의 프로젝트 심층 분석 뷰로 이동
    - External Dev/PM 관점: 클릭 한 번에 해당 프로젝트 대시보드 진입, 핵심 취약점 요약 표시
